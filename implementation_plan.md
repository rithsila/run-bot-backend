# MT5 License Validation (Asymmetric Challenge-Response Protocol)

This plan outlines the architecture and implementation steps to replace the current MT5 license activation flow with a secure, server-side validation system using asymmetric encryption and a challenge-response protocol. Per your request, **Hardware ID (HWID) binding has been completely removed** from this plan.

## User Review Required

> [!WARNING] 
> **Shared Secret Delivery**
> In this challenge-response model, the EA/bridge needs a per-license *shared secret* to sign the challenge, while the server holds the Argon2 hash of this secret. 
> Since we removed HWID, any EA instances holding a valid `licenseKey`, `accountLogin`, and the per-license `sharedSecret` can authenticate.
> **Question**: How do you want to deliver the `sharedSecret` to the user? 
> 1. Generated in the Console Dashboard when the license is created, then the user pastes it into the MT5 EA/Bridge config?
> 2. Auto-generated on the first activation attempt using *just* the license key, then stored securely by the EA locally? (Simpler for the user, but means the first activation determines the secret).

> [!IMPORTANT]
> **API Deprecation**
> The current `@Public()` endpoints in `MembershipsController` (`/memberships/activate` and `/memberships/activate/free`) will remain functional initially to support active instances, but should be phased out once the new EA/Bridge version is rolled out.

### Core Business Logic

#### [NEW] [license-validation.service.ts](file:///Users/rithsila/Projects/bhub-api/src/memberships/license-validation.service.ts)
Create a new service dedicated strictly to the license lifecycle:
- `generateSharedSecret()`: Cryptographically generates a secure, random string (e.g., 32 characters) and returns the plaintext (for the user) and Argon2 hash (for the DB).
- `issueChallenge(licenseKey, accountLogin)`: Generates a 32-byte nonce, stores it in Redis with a 60s TTL, and returns it.
- `verifyAndActivate(dto)`: 
  - Validates `licenseKey` and `accountLogin` exists and is authorized.
  - Retrieves and deletes the Redis nonce to prevent replays.
  - Reconstructs the expected HMAC proof using the retrieved nonce.
  - Verifies the user's signature using `timingSafeEqual`.
  - Determines subscription status.
  - Signs and returns an ECDSA (P-256) JWT using the existing `JoseService`.
- `refreshToken(token, proof)`: Verifies an existing JWT, validates a new HMAC proof, checks if the license `isRevoked`, and issues a new JWT with an extended TTL.

## The Crypto Protocol

1. **Tokens**: Server uses `JoseService` mapped to `SIGNING_PRIVATE_JWK`. The generated token is returned to the EA.
2. **EA Verification**: The EA embeds the public keys defined in `PUBLIC_JWKS`. It verifies the JWT signature offline before accepting it.
3. **Proof Mechanism**: 
   - Nonce = randomly generated by server and cached in Redis.
   - Proof = `HMAC_SHA256(sharedSecret, nonce + licenseKey + accountLogin + serverTimestamp)`.

## Open Questions

1. **Please confirm your preferred delivery mechanism for the `sharedSecret`** (see the alert block at the top of this document).
2. **What should the JWT Token TTL be?** The current JOSE token TTL is 30 days (`TOKEN_TTL_DAYS`). With the introduction of the `/license/refresh` endpoint, we generally reduce the TTL (e.g., to 4 or 8 hours) to ensure rapid remote revocation.
